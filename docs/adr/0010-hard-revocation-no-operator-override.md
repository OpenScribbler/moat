# 0010. Hard revocation without operator override — protocol-level refusal vs MOAT_ALLOW_REVOKED escape hatch

Date: 2026-05-12
Status: Accepted
Feature: npm-distribution-spec
Supersedes: ADR-0007

## Context

ADR-0007 chose to keep an operator-acknowledged escape hatch (`MOAT_ALLOW_REVOKED` with a REQUIRED reason co-variable, per-entry RFC 3339 expiry, and a structured override-applied log event) on the theory that legitimate incident-response workflows occasionally need to materialize a Content Item whose hash has been revoked while the underlying claim is being investigated. The Round-2 form patched the worst failure modes of the Round-1 minimal proposal (no expiry, no reason, no audit anchor) and made the override survivable. Round-3 reviewers (Remy, SpecPurist, Karpathy) converged on a sharper diagnosis: the escape hatch fails MOAT's load-bearing "creators do as little as possible" principle for the *consumer* side of the protocol. A Conforming Client that exposes the override must implement six normative MUSTs (process-scope read-once, REQUIRED reason co-variable, per-entry encoding, RFC 3339 expiry comparison at resolve time, structured override-applied event, per-hash never-global) plus the structured-error path for the malformed-co-variable case — and every one of those MUSTs is a place the implementation can drift from the spec without any external observer noticing. The "works fine without it" test (`CLAUDE.md:125`) fails: if a Conforming Client elects not to implement the override, it MUST treat `MOAT_ALLOW_REVOKED` as if absent — meaning the override is structurally a per-implementation feature, not a protocol guarantee, which is exactly the shape MOAT's enforcement principle (`CLAUDE.md:127`) forbids. The day-one test (`CLAUDE.md:121`) also fails: at ship time, zero Conforming Clients will have implemented the override correctly, so the realistic ecosystem behavior is "operator sets the env var, the install tool ignores it, the operator concludes MOAT is broken" — the spec language about override mechanics generates noise without delivering trust. A simpler design — revocation is hard at the materialization boundary, full stop — has a clean answer to all three tests: the rule is enforceable (refuse to materialize), the absence of any override mechanism is itself observable and trivially conformant, and incident-response workflows route through correcting the authoritative source (lockfile or Registry Manifest) rather than overriding it locally.

## Decision

Chose **Hard revocation at the materialization boundary with no per-hash operator escape hatch (D3a)** over **MOAT_ALLOW_REVOKED with hardened co-variable, per-entry expiry, and structured override-applied logging (ADR-0007's chosen form, C-3); Lockfile-only override via hand-edited `revocation_overrides[]` field (Round 1 design.md option D)**.

## Consequences

The Round-3 sub-spec deletes the entire `## MOAT_ALLOW_REVOKED Operator Override (normative)` section and replaces the §Revocation `Operator-acknowledged proceed (normative — MAY)` clause with a `No operator override (normative — MUST NOT)` clause that explicitly names env vars, command-line flags, and configuration entries as the surfaces a Conforming Client MUST treat as if absent. The resolve-time log entry MUST identify the `source` of the revocation as the literal value `lockfile` or `registry_manifest` so that an operator reviewing logs can route their correction (edit the local lockfile vs decline to sync a Registry Manifest revocation entry) to the right authority. The lockfile-only override alternative was rejected on the same Round-1 grounds — high friction for legitimate incident response — but the friction critique no longer carries the spec, because the friction is now an intentional feature: incident response in MOAT routes through correcting the authoritative source, which is the only path that produces an auditable change of state. Conforming Clients that previously implemented the ADR-0007 form of the override MUST remove the override surface to be Round-3 conformant; ADR-0007 is flipped to `Superseded by ADR-0010` to record the trade-off transition for downstream implementers.
