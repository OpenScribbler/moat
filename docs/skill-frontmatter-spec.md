# Canonical AI Agent Skill Frontmatter Specification

**Status:** Draft
**Date:** 2026-04-06
**Scope:** The SKILL.md frontmatter schema for canonical AI agent skills

---

## Purpose

This specification defines the frontmatter schema for SKILL.md files in the canonical AI agent skill format. It is implementation-agnostic — any tool that reads, writes, converts, or distributes AI agent skills should produce frontmatter conforming to this schema.

This spec intentionally does not define:
- **Provenance or trust** — that is the domain of MOAT (registry protocol)
- **Distribution mechanics** — how skills are packaged, signed, or registered
- **Platform-specific formats** — each platform's native format is defined separately; this is the canonical interchange format
- **Runtime enforcement** — how permissions are enforced at execution time is up to the agent platform

---

## Design Principles

**Behavioral, not historical.** Fields describe what a skill does and what it needs — not where it came from, who wrote it, or what tools happened to exist when it was created. Provenance belongs in the registry layer (MOAT), not the skill file. The evidence for this separation is detailed in the [Why Provenance Belongs in the Registry](#why-provenance-belongs-in-the-registry) section.

**Stable across time.** A skill's permissions and binary requirements will still be meaningful in ten years. Platform names, author identities, and schema versions will not. Fields that age poorly are excluded by design.

**Tooling generates, humans optionally maintain.** Required fields (`name`, `description`) are authored by humans. Optional structured fields (`permissions`, `requires`) should be generated by conforming tools on ingest or conversion — not hand-authored. The spec does not require human provenance workflows.

**Forward-compatible.** Parsers MUST ignore unknown fields. This ensures skills produced against future versions of this spec remain loadable by older implementations without error.

**Compatible with agentskills.io.** The base required fields (`name`, `description`) and optional fields (`license`, `compatibility`) are a superset of the agentskills.io v1 spec. Skills conforming to this spec are valid agentskills.io skills.

## Why Provenance Belongs in the Registry

This section documents the rationale and evidence base for keeping provenance out of SKILL.md frontmatter. It is written for the audience evaluating this spec — the reasoning should be understood, not just accepted.

### The circular hash problem

The most technically decisive argument: a file cannot contain a cryptographic hash of itself.

To compute `content_hash`, you hash the file's bytes. But if the file contains the `content_hash` field, the hash changes when you write the hash into it. You cannot know what to write until after you write it. The workarounds all fail:

- **Blank the field before hashing** — now your hash algorithm has a special case, and every implementation must implement that special case correctly and identically. The adversarial review of a previous version of this spec found this approach introduced multiple divergence points across implementations, including ambiguity about what exact regex to use for blanking, whether leading whitespace is included, and whether the field must use double quotes. Two implementations that handle this differently produce different hashes for identical content.
- **Hash only non-frontmatter content** — then the hash doesn't cover the frontmatter fields that matter (name, description), which defeats the purpose.
- **Put the hash in a sidecar file** — now you have a two-file coordination problem. The sidecar can be deleted, stripped by aggregators, or left out-of-sync when the main file is updated.

MOAT resolves this cleanly: the registry manifest is a separate document that contains the content hash for each item. The hash covers the entire content directory, computed externally. No circular dependency.

### Adoption data shows humans don't fill in optional fields

A corpus analysis of 180,605 SKILL.md files in the wild found:

| Status | Count |
|---|---|
| Valid frontmatter | 175,586 |
| Missing frontmatter entirely | 2,895 |
| Unparseable frontmatter | 2,142 |

This data covers only the **required** fields (`name` and `description`). Approximately 2.8% of skills fail to provide even the minimum. Optional provenance fields — `authors`, `source_repo`, `version`, `license_url` — would have far lower fill rates. A separate study found 642,168 skills with no associated sidecar metadata file, despite the sidecar being recommended.

The conclusion: provenance infrastructure that depends on human authoring will be absent or stale in the vast majority of community-distributed skills. A system that generates false confidence ("provenance present") for the 3-5% of skills where someone filled it in, while silently providing no coverage for the 95-97% where they didn't, is worse than a system that is honest about where provenance lives.

### Unverifiable claims create false confidence

Fields like `authors`, `source_uri`, and `version` in a skill file are unverifiable assertions. There is no mechanism to confirm that the person named in `authors` actually wrote the skill, that the URL in `source_uri` is the actual origin, or that the `version` number reflects real change history.

The ClawHub registry poisoning (Q1 2026) demonstrated this concretely: malicious skills were published with professional-looking documentation, plausible metadata, and accounts with legitimate history. Snyk's ToxicSkills audit found 283 malicious skills (7.1% of the registry at peak infection), including five of the top seven most-downloaded skills. The metadata of those skills was indistinguishable from legitimate content. Five of those top skills had unverifiable but plausible author and source fields.

An LLM reading `authors: "Google Security Team"` in a frontmatter field has no way to verify that claim. Surfacing it to a user as if it were meaningful creates false confidence — the user sees what looks like attribution and lowers their guard. The correct response is to not present unverifiable claims as meaningful metadata.

### Every successful package ecosystem puts trust at the registry boundary

Research across six production package ecosystems (npm, Cargo/crates.io, Go modules, Homebrew, mise, SLSA/in-toto/TUF) found a universal pattern: trust is established at the package or registry boundary, not the individual file level.

- **npm** verifies package tarballs; individual `package.json` `author` fields are unverified strings
- **Cargo** verifies `.crate` archives via crates.io; `Cargo.toml` `authors` is freeform and unenforced
- **Go modules** use a checksum database (sum.golang.org) that records hashes of module zips; individual Go files have no integrity mechanism
- **Homebrew** uses SHA-256 verification of formula downloads; formula `url` fields are unverified strings in the formula file

No production system verifies individual files in isolation. The registry is the trust boundary. MOAT follows this same pattern for AI agent content.

### Content moves without its metadata

Aggregator repositories — which copy skills from one source to another — routinely strip or modify frontmatter without malicious intent. This is a known pattern in the AI skills ecosystem. Whatever provenance metadata is added to a skill file will be incorrect or absent in a significant fraction of the copies that circulate in the wild.

The MOAT approach is honest about this: content that is distributed through a conforming MOAT registry has registry-level attestation. Content that is copied informally has no attestation. The trust tier is visible and accurate, rather than hidden behind frontmatter fields that may be stale or fabricated.

## Schema

### Required Fields

#### `name`

**Type:** string
**Constraints:** Lowercase letters, numbers, and hyphens only. Must start and end with a letter or number. Max 64 characters. Must match the skill's directory name.

**Rationale:** A machine-stable identifier is required for CLI commands, lockfiles, dependency references, and registry lookup. The format restrictions (lowercase, no spaces, alphanumeric boundaries) ensure the identifier is safe for use in file paths, URLs, and shell commands without escaping. The directory name requirement ensures the identifier is consistent with how the skill is stored on disk. The 64-character limit follows the agentskills.io spec and is sufficient for descriptive names without enabling names that are unreadable in terminal output.

```yaml
name: code-review
```

#### `description`

**Type:** string
**Constraints:** Non-empty. Max 1024 characters.

**Rationale:** The description serves two purposes simultaneously. For humans, it explains what the skill does and when to use it. For agent platforms, it is the primary signal used to route tasks to skills — platforms match user requests against skill descriptions to decide which skills to activate. This dual purpose means a good description is both a human-readable summary and a trigger phrase. The 1024-character limit allows enough detail for precise routing without becoming a full documentation page. Skills with poor descriptions are poorly routed — this field is architecture, not just documentation.

```yaml
description: Reviews code for quality, security, and style issues. Use when reviewing pull requests or auditing existing code.
```

### Optional Fields

#### `license`

**Type:** string
**Constraints:** SPDX license identifier preferred (e.g., `MIT`, `Apache-2.0`, `GPL-3.0`). Free-text acceptable for non-SPDX licenses. May be a relative path to a LICENSE file in the skill directory.

**Rationale:** Unlike author or source fields, license information has legal meaning that travels with the content regardless of verification. A user or organization distributing a skill under a license they haven't complied with has real legal exposure — and knowing the license is actionable information even without cryptographic proof. SPDX identifiers are machine-readable, enabling automated license compliance checks. This field is the one provenance-adjacent field that belongs in the content file because its value is meaningful even without verification. Already part of the agentskills.io spec.

```yaml
license: MIT
```

#### `compatibility`

**Type:** string
**Constraints:** Max 500 characters. Free text.

**Rationale:** A free-text escape valve for environmental requirements that don't fit the structured `requires` field — runtime platform requirements, account dependencies, network access requirements, or tool-specific constraints. This field is for human readers; it is not machine-enforced. Included for agentskills.io compatibility and to handle edge cases not covered by `requires`. When a requirement can be expressed in `requires`, prefer `requires` — it is more actionable.

```yaml
compatibility: Requires a GitHub account with repository access. Designed for use with TypeScript projects.
```

#### `permissions`

**Type:** object
**Generated by:** Conforming tools on ingest or conversion. May be authored manually.

**Rationale:** The `permissions` block is the most significant addition beyond the agentskills.io base spec. It serves two audiences:

For **agent platforms**: A machine-readable declaration of what system access this skill instructs the agent to use. Platforms can use this to surface risk at install time, gate skill activation behind user confirmation, or implement governance policies without human review of every skill. The [OWASP Agentic Skills Top 10](https://github.com/OWASP/www-project-agentic-skills-top-10) — specifically [AST09 (No Governance)](https://github.com/OWASP/www-project-agentic-skills-top-10/blob/main/ast09.md) and [AST10 (Cross-Platform Reuse)](https://github.com/OWASP/www-project-agentic-skills-top-10/blob/main/ast10.md) — specifically recommends `risk_tier` for automated governance — `permissions` is the source data from which risk tier is derived.

For **LLMs loading the skill**: An LLM that reads `shell: false` in the frontmatter has context that this skill does not expect to run shell commands. This is actionable context — if the skill later attempts to instruct shell execution, the LLM has a signal that something is anomalous.

Unlike provenance fields, `permissions` describes behavior that is inherent to the skill and verifiable by content analysis. A conforming tool can read the skill body and determine whether it instructs file writes, shell execution, or network access — then generate the `permissions` block automatically. This aligns with the "tooling generates, humans don't need to" principle.

When absent, no permissions are declared. Conforming tools SHOULD treat absent `permissions` as unknown, not as unrestricted.

```yaml
permissions:
  files:
    read:
      - "**/*.go"           # glob patterns — omit for unrestricted
    write:
      - "src/**"
    deny_write:
      - "AGENTS.md"         # files this skill explicitly will not write
      - "MEMORY.md"
  network:
    allow:
      - "api.github.com"    # domain allowlist
    deny: "*"               # deny all not in allow list
  shell: false              # whether skill instructs shell command execution
```

**Sub-fields:**

| Field | Type | Description |
|---|---|---|
| `files.read` | string[] | Glob patterns for files the skill reads. Omit for unrestricted. |
| `files.write` | string[] | Glob patterns for files the skill writes. Omit for unrestricted. |
| `files.deny_write` | string[] | Files this skill explicitly never writes. |
| `network.allow` | string[] | Allowed domains. Omit for unrestricted. |
| `network.deny` | string | `"*"` to deny all not in allowlist. |
| `shell` | boolean | Whether skill instructs shell command execution. |

All sub-fields are optional. A partial `permissions` block is valid.

#### `requires`

**Type:** object
**Generated by:** Conforming tools on ingest or conversion. May be authored manually.

**Rationale:** Unlike `compatibility` (free text), `requires` is structured and machine-actionable. A conforming tool can check whether `ripgrep` is installed before completing installation of a skill that requires it, surface a clear warning if it is not, and give the user a concrete list of what to install. This is directly verifiable — binary presence is a checkable fact, not an assertion. Version constraints using semver syntax enable tools to check not just presence but compatibility.

The `programming_environments` sub-field covers language runtimes, which are similarly checkable. This replaces the free-text `compatibility` field for the most common class of environment dependency.

Conforming tools SHOULD warn (but not block by default) when `requires` dependencies are absent. Blocking install on a missing binary would prevent installation in cases where the binary exists but is not in PATH, or where the skill is being installed for a different environment than the installing machine.

```yaml
requires:
  binaries:
    - name: ripgrep
      version: ">=14.0"     # optional version constraint
    - name: jq
  programming_environments:
    - name: node
      version: ">=20"
    - name: python
      version: ">=3.11"
```

**Sub-fields:**

| Field | Type | Description |
|---|---|---|
| `requires.binaries` | object[] | Command-line tools that must be installed. Each entry has `name` (required) and `version` (optional, semver constraint). |
| `requires.programming_environments` | object[] | Language runtimes required. Each entry has `name` (required) and `version` (optional, semver constraint). |

Version constraints follow semver constraint syntax: `>=1.0.0`, `~>1.2`, `==1.0.0`.

## What Is Explicitly Excluded

The following fields were considered and rejected. They are documented here — with rationale — so future spec discussions do not relitigate them without new evidence.

### Provenance fields (`source_uri`, `authors`, `version`, `content_hash`, `signature`)

**Why excluded:** See [Why Provenance Belongs in the Registry](#why-provenance-belongs-in-the-registry) for the full evidence base. Summary:

- `content_hash` cannot be placed in the file it hashes without introducing implementation-divergent workarounds (circular dependency problem)
- `signature` requires consumers to locate and verify a public key — a problem no production ecosystem has solved for arbitrary community publishers
- `source_uri` and `authors` are unverifiable claims that create false confidence without cryptographic proof
- `version` as a human-maintained field will be stale or missing in the vast majority of community skills (corpus evidence: 2.8% failure rate on required fields alone); version tracking belongs in the registry manifest where tooling manages it

These fields belong in the MOAT registry manifest, where they are generated by tooling, cryptographically signed, and verified by conforming clients.

### Schema versioning (`skill_metadata_schema_version`, `skill_metadata_schema_definition`)

**Why excluded:** Schema version fields require a published, stable schema URL to be meaningful. Without a URL that resolves to a machine-readable schema, the field is noise. Without enforcement, schema version gating creates implementation divergence (implementations that reject unrecognized versions vs. implementations that accept them) with no benefit. The correct approach is forward-compatible parsing: parsers ignore unknown fields, older parsers load newer skills without error, and the spec itself is versioned via its publication history.

### Platform declarations (`platforms`, `supported_agent_platforms`)

**Why excluded:** Declaring platform compatibility in a skill file encodes a snapshot of which platforms exist at the time of writing. This creates two problems:

First, tools that convert skills between platforms (which is the primary use case for a canonical format) make these declarations immediately stale. A skill marked `platforms: [claude-code]` that is converted for use on Cursor is now inaccurate. The field describes history, not capability.

Second, platform names are ephemeral. The platforms named in a 2026 spec (Claude Code, Cursor, Windsurf) will be as meaningful in 2036 as "Sublime Text plugin" or "Atom package" are today. A field that requires updating as the platform landscape evolves is a maintenance liability, not a feature.

Platform affinity is a registry and implementation concern — registries know which platforms they serve, and tools know which formats they support. This information does not belong in the content file.

### Date fields (`created_at`, `last_modified_at`, `published_at`)

**Why excluded:** Human-maintained date fields become inaccurate when content is updated without updating the date — which is common. A `last_modified_at` date that reflects a typo fix from 18 months ago is actively misleading for a skill whose instructions are current. The date creates a false impression of staleness.

Tooling-generated dates belong in the registry manifest (`attested_at`), where they can be reliably computed at attestation time and updated on re-attestation. The registry date accurately reflects when a registry last vouched for the content — which is the meaningful staleness signal — without requiring the content file to be modified.

### Services declarations (`expectations.services`)

**Why excluded:** A field declaring "this skill requires a GitHub account" or "this skill requires a Notion subscription" is not machine-verifiable. Tools cannot check whether the user has a valid account with an external service. The information is useful for human readers but belongs in the `compatibility` free-text field, where it is clearly advisory rather than structured. Adding structure to an unverifiable field implies machine-actionability that does not exist.

## Full Example

```yaml
---
name: code-review
description: Reviews code for quality, security, and style issues. Use when reviewing pull requests or auditing existing code.
license: MIT
compatibility: Works best with TypeScript and Go projects.
permissions:
  files:
    read:
      - "**/*.go"
      - "**/*.ts"
      - "**/*.tsx"
    write: []
    deny_write:
      - "AGENTS.md"
      - "MEMORY.md"
  network:
    deny: "*"
  shell: false
requires:
  binaries:
    - name: ripgrep
      version: ">=14.0"
---

# Code Review

Reviews code for quality issues, security vulnerabilities, and style consistency.

## When to use this skill

Use when asked to review a pull request, audit existing code, or check a file for issues before committing.
```

## Conformance

A SKILL.md file conforms to this specification if:
- `name` and `description` are present and valid
- All optional fields, when present, match the types and constraints defined above
- Unknown fields do not cause the file to be rejected

A conforming parser MUST:
- Accept files with unknown top-level fields without error
- Not reject files that omit any optional field
- Not infer permissions or requirements from field absence

A conforming tool that generates or modifies SKILL.md files SHOULD:
- Populate `permissions` and `requires` based on analysis of skill content
- Not populate provenance fields (`source_uri`, `authors`, `content_hash`, etc.) in SKILL.md — these belong in the MOAT registry manifest
- Preserve existing `permissions` and `requires` values on round-trip unless they are known to be incorrect
